# 电商热点情报系统项目方案

> 方案版本：2026-06-13  
> 项目仓库：`caishenyeweifanchi-cmyk/ecommerce-news-radar`  
> 参考项目：`LearnPrompt/ai-news-radar`  
> 当前定位：从 AI 新闻雷达改造成电商运营可用的情报系统

---

## 1. 一句话定位

本项目不是“泛新闻聚合站”，也不是“AI 新闻站”。

它应该被定义为：

```text
电商热点情报系统：
追踪平台规则、内容电商玩法、投流素材变化、AI 能力、选品趋势和与你业务相关的运营机会，
并把原始信息转化成可判断、可点击、可行动的运营情报。
```

核心判断标准只有一个：

```text
这条信息能不能帮助电商运营、内容生产、投流、选品、店铺经营或自动化提效？
```

回答不了这个问题的信息，可以进入原始池或观察池，但不能进首页、精选和日报。

## 2. 当前项目诊断

### 2.1 当前可用基础

项目已经具备这些基础能力：

- 静态网页：`index.html` + `assets/app.js` + `assets/styles.css`
- 定时刷新：GitHub Actions 定时执行采集脚本
- RSS/OPML 采集：`feeds/ecommerce.example.opml`
- 重点网页源配置：`feeds/ecommerce.web-sources.json`
- AI 候选源配置：`feeds/ai-commerce.candidates.json`
- 数据输出：`data/*.json`
- 规则评分：`scripts/ecommerce_relevance.py`
- 主采集流程：`scripts/update_news.py`
- 测试：`tests/*.py`

### 2.2 当前主要问题

当前问题不是单一 bug，而是产品边界没有收住：

1. 页面像新闻看板，不像电商运营情报台。
2. 数据源数量看起来很多，但真正能稳定产出“详情页 + 真实时间”的源少。
3. RSS、网页源、AI 源、电商媒体源混在一起，缺少清晰分层。
4. 首页、日报、AI 情报、电商动态之间的入选规则不够明确。
5. 一些按钮曾经像摆设，搜索、频道筛选、导航必须逐个验收。
6. “当前热点”和“时间线”应该同时存在，不能互相替代。
7. 活动时间、报名时间、网页抓取时间不能当作发布时间。
8. 官方平台规则必须优先，但只能在抓到详情页和真实发布时间后进入日报。
9. AI 信息源不能卡得太死，但必须区分“纯 AI 情报”和“可转电商的 AI 能力”。

### 2.3 当前数据状态参考

最近一次刷新后的状态：

- `feeds/ecommerce.web-sources.json`：1048 个候选网页源，当前启用 180 个，P0 源 160 个。
- `feeds/ai-commerce.candidates.json`：28 个 AI 候选源。
- `data/source-audit.json`：171 个审计源，其中建议启用 19 个，禁用 5 个。
- `data/latest-24h.json`：当天强相关 6 条。
- `data/latest-24h-all.json`：窗口内全量候选 91 条。
- `data/ai-radar.json`：AI 情报 439 条。
- `data/daily-brief.json`：当天日报 1 条。
- `data/stories-merged.json`：当天故事线 6 条。

这个状态说明：采集广度已经有了，但可进入首页和日报的高质量信息偏少。后续重点不是盲目加源，而是提高“可采详情页 + 真实时间 + 电商价值解释”的比例。

## 3. 产品结构设计

### 3.1 左侧导航

最终左侧导航建议固定为：

```text
精选
今日必看
平台规则
运营玩法
AI 情报
AI 电商
投流素材
电商动态
电商日报
信源提报
关于
```

删除或不再暴露：

```text
更新日志
反馈
扩展观察
泛新闻式全量动态
没有真实功能的入口
```

### 3.2 页面模块职责

| 模块 | 作用 | 入选规则 |
| --- | --- | --- |
| 精选 | 当前最值得看的高价值事件 | 当天、强相关、高分、多源或高优先级 |
| 今日必看 | 对运营有直接影响的当天内容 | 平台规则、风险、投流变化、强 AI 电商能力 |
| 平台规则 | 官方规则、处罚、准入、履约、售后 | 必须详情页 + 真实发布时间 |
| 运营玩法 | 内容电商、直播、店播、达人、选品、案例 | 必须能解释适合谁、怎么用 |
| AI 情报 | 纯 AI 动态 | 不强行要求电商相关 |
| AI 电商 | AI 能力转电商场景 | 商品图、视频、脚本、客服、选品、投流、自动化 |
| 投流素材 | 广告、素材、投放、达人合作 | 对素材或投放动作有影响 |
| 电商动态 | 行业、电商媒体、跨境、物流、支付 | 默认不抢首页，重要时升级 |
| 电商日报 | 当天摘要 | 当天、可点击、有影响、有建议动作 |
| 信源提报 | 用户提交信息源 | 进入候选池，不直接进首页 |
| 关于 | 项目说明、联系方式、数据规则 | 不放假功能 |

### 3.3 当前热点与时间线

两者必须同时保留：

- 当前热点：按分数、多源重复、来源等级、当天影响排序。
- 时间线：按真实发布时间排序。

规则：

```text
当前热点不是时间线的替代品。
时间线不是当前热点的替代品。
当前热点负责“先看什么”。
时间线负责“什么时候发生”。
```

没有多源重复时，当前热点仍然展示高分内容，不应该空白。

## 4. 数据分层设计

项目必须从“一个池子混筛”改成清晰数据分层。

### 4.1 数据层级

```text
raw_items
  原始采集结果，保留真实标题、链接、发布时间、来源。

ai_radar
  纯 AI 情报池，允许更宽，不要求直接电商相关。

ecommerce_candidates
  电商候选池，经过电商相关性评分。

stories
  事件合并后的故事线，合并同一事件的多个报道。

daily_brief
  当天日报，只保留当天、可点击、对运营有用的信息。
```

### 4.2 输出文件职责

| 文件 | 职责 | 是否面向首页 |
| --- | --- | --- |
| `data/latest-24h.json` | 当天强相关电商信息 | 是 |
| `data/latest-24h-all.json` | 当前窗口内全量候选 | 部分，用于电商动态/调试 |
| `data/ai-radar.json` | 纯 AI 情报 | 是，用于 AI 情报板块 |
| `data/stories-merged.json` | 当天事件故事线 | 是 |
| `data/daily-brief.json` | 当天日报 | 是 |
| `data/source-status.json` | 信息源运行状态 | 底部/关于，不占主视觉 |
| `data/source-audit.json` | 信息源质量审计 | 管理/维护使用 |
| `data/archive.json` | 历史归档 | 不直接展示 |

### 4.3 每条信息必须具备的字段

基础字段：

```text
title
url
source
site_id
published_at
first_seen_at
topic_label
topic_score
```

电商解释字段：

```text
impact
suggested_action
business_value
operational_depth.operator_question
operational_depth.affected_roles
operational_depth.action_priority
operational_depth.why_it_matters
operational_depth.next_action
```

网页源质量字段：

```text
meta.web_source_priority
meta.web_source_domain
meta.web_source_mode
meta.web_source_url
meta.article_url_source
meta.published_time_source
```

### 4.4 时间规则

时间是项目的硬规则。

```text
日报、今日必看、当前热点、平台规则首页展示，只允许北京时间当天信息。
```

有效发布时间来源：

```text
api
detail_meta
detail_text
list_text
```

无效时间来源：

```text
snapshot
unconfirmed
first_seen_at
last_seen_at
报名时间
活动开始时间
活动结束时间
页面抓取时间
```

活动类信息可以保留活动时间，但必须单独字段展示，不能写进发布时间。

## 5. 信息源策略

### 5.1 信息源分级

| 等级 | 类型 | 示例 | 默认用途 |
| --- | --- | --- | --- |
| P0 | 官方规则/公告/投流规则 | 抖音电商、小红书电商、快手电商、淘宝天猫、京东、千川、聚光 | 可进首页和日报 |
| P1 | 高质量电商媒体/运营媒体 | 电商报、亿邦、派代、新榜、运营研究社、见实、品牌星球 | 可进首页，择优进日报 |
| P2 | AI/工具/案例源 | OpenAI、Claude、Gemini、DeepSeek、AI 视频、AI 图片、Agent | 进 AI 情报，可迁移才进 AI 电商 |
| P3 | 观察源 | 跨境论坛、Reddit、Amazon、Shopify、TikTok Shop、物流支付 | 默认进电商动态，不抢首页 |

### 5.2 源池原则

不要追求表面数量。

```text
候选源可以多。
启用源要稳。
首页源要准。
日报源要严格。
```

源状态必须分清：

```text
真实采集
快照监控
失败
候选未启用
需要适配器
需要登录，不接入公共版本
```

### 5.3 官方规则源策略

官方规则源必须优先开发适配器：

```text
抖音电商规则中心
小红书电商官方资讯
快手电商规则中心
淘宝规则中心
天猫规则频道
京东商家规则中心
阿里妈妈规则中心
巨量千川规则中心
巨量引擎帮助中心
腾讯广告帮助中心
微信小店
拼多多规则
```

进入首页和日报的最低标准：

```text
必须拿到规则详情页 URL。
必须拿到真实发布时间。
标题必须是具体规则/公告，不是规则中心首页。
```

如果只能拿到规则中心首页：

```text
只进入源状态。
不能进首页。
不能进日报。
不能伪装成规则更新。
```

### 5.4 AI 情报源策略

AI 信息源不能卡得太死。

正确方式：

```text
先广泛采集 AI 信息。
再判断是否能迁移到电商。
能迁移的进入 AI 电商、今日必看、日报。
不能迁移的只进入 AI 情报。
```

允许进入 AI 电商的能力：

```text
商品图生成
AI 视频
广告素材
直播切片
短视频脚本
多模态页面分析
评论分析
竞品分析
客服自动化
选品分析
投流复盘
Agent 浏览器能力
表格/飞书自动化
成本下降
速度提升
长上下文运营诊断
```

不能直接这样展示：

```text
OpenAI 发布新模型
```

应该转成：

```text
OpenAI 新模型能力变化｜可用于批量生成商品图/广告素材/短视频脚本｜适合素材测试提效
```

## 6. 评分与筛选规则

### 6.1 三层门槛

第一层：场景相关。

必须命中至少一个方向：

```text
平台规则
账号风险
商品治理
售后履约
直播带货
短视频带货
种草
达人
店播
投流
广告素材
选品
爆款
客服
私域
AI 提效
自动化
跨境卖家经营
物流/支付/关税
```

第二层：业务价值。

每条信息必须归到一个价值类型：

```text
规则风险
运营玩法
AI 提效
投流素材
选品趋势
平台机会
电商动态
```

第三层：质量拦截。

明确屏蔽：

```text
栏目页
首页
无真实发布时间的网页源
纯活动报名时间冒充发布时间
泛科技
泛融资
资本市场
医药
汽车
普通政策
无电商场景的模型新闻
泛创业社区闲聊
低价值论坛吐槽
```

### 6.2 首页入选规则

首页只展示：

```text
P0/P1 高质量电商信息
可迁移到电商场景的 P2 AI 信息
少量高分 P3 观察信息
```

首页不展示：

```text
源状态
抓取失败
候选源
栏目页
快照页
无真实时间信息
```

### 6.3 日报入选规则

日报比首页更严格：

```text
必须是北京时间当天。
必须可点击跳转。
必须回答“对电商运营有什么用”。
必须有建议动作。
官方规则优先。
无真实发布时间的网页源不进日报。
```

## 7. 日报设计

日报结构固定为：

```markdown
# 电商热点日报

## 今日必看
- 平台｜标题｜影响｜建议动作｜链接

## 平台规则与风险
- 平台｜规则变化｜影响对象｜风险点｜链接

## 运营玩法与热点
- 玩法｜适合谁｜可以怎么用｜链接

## AI 电商
- AI能力/工具｜可用于什么电商场景｜建议动作｜链接

## 投流与素材
- 平台｜变化｜对素材/投放的影响｜链接

## 电商动态
- 跨境/物流/支付/行业｜是否重要｜为什么｜链接
```

每条日报必须包含：

```text
标题
来源
链接
真实发布时间
频道
影响
建议动作
优先级
```

日报不是越多越好。宁可 3 条高质量，也不要 20 条垃圾。

## 10. 分阶段实施方案

### 阶段 0：冻结方案和验收标准

目标：

```text
先停止无序补丁式开发，确认产品结构、数据规则、信息源策略和验收标准。
```

要做：

1. 确认本方案作为后续开发依据。
2. 把 README 里旧 AI 新闻雷达和乱码内容重写。
3. 给 Claude / Quest / 其他 Agent 的协作边界写入文档。

验收：

```text
docs/ECOMMERCE_INTELLIGENCE_RADAR_PROJECT_PLAN.md 存在。
README 与项目定位一致。
其他 Agent 知道只能按指定文件范围修改。
```

### 阶段 1：数据规则收敛

目标：

```text
把数据层从“混合新闻池”收敛成“原始池、AI 情报、电商候选、故事线、日报”。
```

要改：

- `scripts/update_news.py`
- `scripts/ecommerce_relevance.py`
- `tests/test_daily_brief.py`
- `tests/test_ecommerce_relevance.py`

验收：

```text
日报只含北京时间当天。
活动报名时间不会进入发布时间字段。
网页源无详情页或无真实时间不能进日报。
每条首页信息都有 operational_depth。
pytest 全部通过。
```

### 阶段 2：页面交互清理

目标：

```text
所有按钮都有作用，没有摆设入口。
```

要改：

- `index.html`
- `assets/app.js`
- `assets/styles.css`

验收：

```text
左侧导航每一项都能跳到对应区域。
搜索框能实时过滤。
频道筛选能真实改变列表。
当前热点和时间线都保留。
日报条目可点击跳转。
更新日志和反馈不再出现。
扩展观察改为电商动态。
```

### 阶段 3：官方规则源适配器

目标：

```text
优先让 P0 官方平台规则源抓到详情页和真实发布时间。
```

优先级：

1. 抖音电商规则中心
2. 小红书电商官方资讯
3. 快手电商规则中心
4. 京东商家规则中心
5. 淘宝/天猫规则
6. 阿里妈妈
7. 巨量千川/巨量引擎
8. 腾讯广告/微信小店

验收：

```text
每个适配器至少有测试。
输出 URL 是详情页，不是中心页。
published_time_source 是 api/detail_meta/detail_text/list_text。
source-audit 能标记源是否可进首页。
```

### 阶段 4：AI 情报与 AI 电商分离

目标：

```text
AI 情报足够广，AI 电商足够准。
```

要做：

1. 从 `LearnPrompt/ai-news-radar` 合理迁移 AI 信息源。
2. 扩充官方 AI、模型、Agent、AI 视频、AI 图片、AI 自动化源。
3. AI 信息先进入 AI 情报。
4. 能转电商场景的再进入 AI 电商、今日必看和日报。

验收：

```text
AI 情报页有足够内容。
AI 电商页不出现纯模型新闻标题党。
每条 AI 电商信息都能说明对应电商场景。
```

### 阶段 5：信源治理

目标：

```text
候选源多，启用源准，首页源干净。
```

要做：

1. 清理 1048 个网页候选源。
2. P0/P1 默认启用，P2/P3 分场景启用。
3. 把论坛、跨境、AI、电商媒体、官方规则分组。
4. `source-audit` 输出可读的源质量结果。

验收：

```text
源状态能看出哪些是真采集、哪些是快照、哪些失败。
候选源不会直接污染首页。
论坛源不会霸屏。
跨境源默认只进电商动态。
```

### 阶段 6：日报产品化

目标：

```text
日报可以直接给 Agent、人或飞书工作流读取。
```

要做：

1. `daily-brief.json` 字段稳定。
2. 日报页面可读。
3. 每条日报可点击。
4. Agent 接入说明清楚。

验收：

```text
日报不是标题列表，而是影响 + 建议动作。
平台官方新规优先。
无当天内容时明确显示“今日暂无高优先级情报”，不乱塞旧内容。
```

验收：

笔记本屏幕第一屏能看到有效内容。
卡片信息密度合理。
