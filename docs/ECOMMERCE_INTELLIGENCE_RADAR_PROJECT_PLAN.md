## 1. 产品定位

本项目不做泛新闻聚合，也不做单纯 AI 新闻站。

最终定位：

```text
电商热点情报系统：
追踪平台规则、内容电商玩法、投流素材变化、AI 能力、选品趋势和与你业务相关的运营机会，
并把原始信息转化成可判断、可点击、可行动的运营情报。
```

唯一核心判断标准：

```text
这条信息能不能帮助电商运营、内容生产、投流、选品、店铺经营或自动化提效？
```

回答不了这个问题的信息，可以进入原始池、AI 情报池或电商动态池，但不能进入首页精选、今日必看和日报。

## 3. 产品结构

### 3.1 左侧导航

最终导航：

```text
精选
今日必看
平台规则
运营玩法
AI 情报
AI 电商
投流素材
电商动态
电商日报
信源提报
关于
```

删除或隐藏：

```text
更新日志
反馈
扩展观察
WaytoAGI 等旧 AI 项目残留入口
没有真实功能的按钮
```

说明：

- `AI 情报`：只收集 AI 动态，不强行要求电商相关。
- `AI 电商`：从 AI 情报中筛出能用于电商的内容。
- `电商动态`：普通电商资讯、跨境、物流、支付、论坛、行业观察。
- `精选`：多源重复、高分、当天、强业务价值。
- `今日必看`：当天必须处理或值得立即看的运营情报。

### 3.2 当前热点与时间线

这两个视图必须同时保留：

```text
当前热点：解决“先看什么”。
时间线：解决“什么时候发生”。
```

规则：

- 当前热点按分数、多源重复、来源等级、业务影响排序。
- 时间线按真实发布时间排序。
- 没有多源重复时，当前热点仍按评分展示，不允许空白。
- 时间线不能被删掉，也不能被当前热点替代。

## 4. 数据模型

### 4.1 数据分层

```text
raw_items
  原始采集结果，保留真实标题、真实链接、真实发布时间、来源。

ai_radar
  纯 AI 情报池，允许更宽，不要求直接电商相关。

ecommerce_candidates
  电商候选池，经过电商相关性评分。

stories
  事件合并后的故事线，合并同一事件的多个报道。

daily_brief
  当天日报，只保留当天、可点击、对运营有用的信息。
```

### 4.2 输出文件职责

| 文件 | 职责 | 页面用途 |
| --- | --- | --- |
| `data/latest-24h.json` | 当天强相关电商信息 | 首页、今日必看 |
| `data/latest-24h-all.json` | 当前窗口全量候选 | 电商动态、调试 |
| `data/ai-radar.json` | 纯 AI 情报 | AI 情报 |
| `data/stories-merged.json` | 当天事件故事线 | 当前热点、时间线 |
| `data/daily-brief.json` | 当天日报 | 电商日报 |
| `data/source-status.json` | 信息源运行状态 | 关于/底部 |
| `data/source-audit.json` | 信息源质量审计 | 维护使用 |
| `data/archive.json` | 历史归档 | 不直接展示 |

### 4.3 字段原则

一期前端和日报只强制承诺两个业务字段：

```text
impact            影响谁、影响什么经营动作
suggested_action  建议下一步怎么处理
```

已有代码里的 `operational_depth` 可以作为内部辅助字段保留，但它不是一期前端必须完整渲染的产品承诺。

不再把以下 5 个子字段作为一期强制承诺：

```text
operator_question
affected_roles
action_priority
why_it_matters
next_action
```

后续如果要扩展，优先只新增：

```text
affected_roles
```

扩展条件：

- 前端明确展示。
- 日报明确使用。
- 测试覆盖字段存在和含义。
- 不重复 `impact` 和 `suggested_action`。

### 4.4 发布时间规则

日报、今日必看、当前热点、平台规则首页展示，只允许北京时间当天信息。

有效发布时间来源：

```text
api
detail_meta
detail_text
list_text
```

无效时间来源：

```text
snapshot
unconfirmed
first_seen_at
last_seen_at
页面抓取时间
活动开始时间
活动结束时间
报名时间
```

活动类信息可以保留活动时间，但必须作为活动字段展示，不能写入 `published_at`。

## 5. 信息源策略

### 5.1 信息源分级

| 等级 | 类型 | 默认用途 |
| --- | --- | --- |
| P0 | 官方规则、官方公告、投流规则 | 可进首页和日报，但必须满足详情页 + 真实时间 |
| P1 | 高质量电商媒体、运营媒体 | 可进首页，择优进日报 |
| P2 | AI、工具、案例源 | 进 AI 情报，可迁移才进 AI 电商 |
| P3 | 跨境、论坛、物流、支付、观察源 | 默认进电商动态，不抢首页 |

### 5.2 源池原则

```text
候选源可以多。
启用源要稳。
首页源要准。
日报源要严格。
```

源状态必须分清：

```text
真实采集
快照监控
失败
候选未启用
需要 headless 浏览器
需要登录，不接入公共版本
RSS 代理覆盖
```

### 5.3 P0 官方规则源决策：headless vs RSS 代理

这是阶段 3 的关键决策。

已验证情况：

```text
淘宝、天猫、京东、拼多多、抖音、快手、小红书等规则中心大量依赖 JS 渲染。
标准 HTTP + BeautifulSoup 适配器经常只能拿到规则中心首页，拿不到详情页和真实发布时间。
```

当前决策：

```text
在默认 GitHub Actions 公共版本中，暂不上 Playwright/Selenium 作为主采集路径。
在拥有稳定 headless 浏览器能力之前，P0 官方规则源的覆盖优先通过行业媒体 RSS/政策频道代理，例如亿邦动力政策频道。
JS 渲染的 P0 官方源暂时保留为“快照监控/源健康监控”，不进首页、不进日报。
```

原因：

- Playwright/Selenium 会显著增加 CI 资源、运行时间和维护成本。
- 国内平台 JS 页面结构变动快，headless 适配器容易反复坏。
- 对公共 GitHub Pages 项目，稳定性优先于“看起来什么都抓”。
- 行业媒体 RSS 虽然不是第一手官方页，但能作为规则变化代理信号。

后续升级条件：

```text
只有当某个 P0 官方源能稳定拿到：
1. 详情页 URL
2. 真实发布时间
3. 标题
4. 规则分类
5. 可测试的适配器
才允许进入首页和日报。
```

headless 浏览器只作为二期专项，不作为当前默认路径。

### 5.4 AI 情报源策略

AI 信息源不要卡得太死。

正确流程：

```text
先广泛采集 AI 信息。
再判断是否能迁移到电商。
能迁移的进入 AI 电商、今日必看、日报。
不能迁移的只进入 AI 情报。
```

允许进入 AI 电商的场景：

```text
商品图生成
AI 视频
广告素材
直播切片
短视频脚本
多模态页面分析
评论分析
竞品分析
客服自动化
选品分析
投流复盘
Agent 浏览器能力
飞书/表格自动化
成本下降
速度提升
长上下文运营诊断
```

不能直接展示成：

```text
OpenAI 发布新模型
```

应该转成：

```text
OpenAI 新模型能力变化｜可用于批量生成商品图/广告素材/短视频脚本｜适合素材测试提效
```

### 6.1 三层门槛

第一层：场景相关。

至少命中一个方向：

```text
平台规则
账号风险
商品治理
售后履约
直播带货
短视频带货
种草

投流
广告素材
选品
爆款
客服
私域
AI 提效
自动化
跨境卖家经营
物流/支付/关税
```

第二层：业务价值。

必须归到一个价值类型：

```text
规则风险
运营玩法
AI 提效
投流素材
选品趋势
平台机会
电商动态
```

第三层：质量拦截。

明确屏蔽：

```text
栏目页
首页
无真实发布时间的网页源
纯活动报名时间冒充发布时间
泛科技
泛融资
资本市场
医药
汽车
普通政策
无电商场景的模型新闻
泛创业社区闲聊
低价值论坛吐槽
```

### 6.2 首页入选规则

首页只展示：

```text
P0/P1 高质量电商信息
可迁移到电商场景的 P2 AI 信息
少量高分 P3 观察信息
```

首页不展示：

```text
源状态
抓取失败
候选源
栏目页
快照页
无真实时间信息
```

### 6.3 日报入选规则

日报比首页更严格。

必须满足：

```text
北京时间当天
可点击跳转
有 impact
有 suggested_action
官方规则优先
网页源必须详情页 + 真实发布时间
```

当天高质量内容不足时：

```text
不降格补低质量内容。
不拿昨天内容补今天日报。
不把观察源硬塞进日报。
```

如果当天高优先级内容少于 3 条，日报应该显示：

```text
今日高优先级情报不足，仅保留已验证内容。
```

如果当天 0 条，日报显示：

```text
今日高优先级情报暂无。
```

## 7. 日报设计

日报结构：

```markdown
# 电商热点日报

## 今日必看
- 平台｜标题｜影响｜建议动作｜链接

## 平台规则与风险
- 平台｜规则变化｜影响对象｜风险点｜链接

## 运营玩法与热点
- 玩法｜适合谁｜可以怎么用｜链接

## AI 电商
- AI能力/工具｜可用于什么电商场景｜建议动作｜链接

## 投流与素材
- 平台｜变化｜对素材/投放的影响｜链接

## 电商动态
- 跨境/物流/支付/行业｜是否重要｜为什么｜链接
```

每条必须包含：

```text
标题
来源
链接
真实发布时间
频道
impact
suggested_action
```

日报不是越多越好。宁可 1 条真有用，也不要 20 条垃圾。

## 10. 分阶段实施路线

### 阶段 0：方案冻结和文档对齐

目标：

```text
停止无序补丁式开发，先把产品结构、数据规则、信息源策略和验收标准固定。
```

任务：

1. 以本文档作为主方案。
2. 重写 README，去掉旧 AI News Radar 和乱码残留。
3. 给 Claude / Quest / 其他 Agent 写清楚协作边界。

验收：

```text
README 与本项目定位一致。
其他 Agent 能按方案知道该改什么、不该改什么。
```

### 阶段 1：数据规则收敛

目标：

```text
保证首页、今日必看、日报只展示当天、有用、可点击的信息。
```

任务：

1. 保持当天过滤。
2. 保持网页源详情页 + 真实发布时间门槛。
3. 让 `impact` 和 `suggested_action` 成为前端和日报的主字段。
4. 不再把完整 `operational_depth` 当作一期前端承诺。

验收：

```text
日报只含北京时间当天信息。
活动时间不会进入发布时间字段。
网页源无详情页或无真实时间不能进日报。
每条日报都有 impact 和 suggested_action。
pytest 全部通过。
```

### 阶段 2：页面交互清理

目标：

```text
所有按钮都有作用，没有摆设入口。
```

任务：

1. 清理左侧导航。
2. 去掉更新日志、反馈、扩展观察旧入口。
3. 搜索框真实过滤。

## 11. 多 Agent 协作规则

### 11.1 Codex / Claude / 主开发共同负责

Codex 和 Claude 都属于开发执行角色，可以共同负责：

```text
写测试

整理 README
修样式
数据刷新
本地浏览器验证
GitHub 提交推送
方案维护
```

不得擅自：

```text
删除信息源
改 GitHub Actions 定时任务
改数据输出字段名
放宽日报质量门槛
把栏目页塞进日报
把无发布时间信息写成当天新闻
默认引入 Playwright/Selenium
```

### 11.2 Quest 可负责

```text

补充候选源

```

Quest 交付边界：

```text
只能写入指定信息源清单文件。
不能改采集代码。
不能改前端。
不能改 data/*.json。
不能伪造 RSS。
不能把平台官网首页当详情页。
```

### 11.3 协作分工原则

Codex 与 Claude 的分工按任务现场决定，不固定死：

```text
谁在当前会话里执行，谁负责读方案、改代码、跑测试、说明验证结果。

```
